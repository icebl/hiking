# 模块细化：轨迹记录

> 所属：徒步APP PRD · 第 3.1 节展开
> 版本：v0.1 · 2026-06-24
> 定位：APP 最核心、状态最多的模块。**底线是绝不丢轨迹。**

---

## 1. 这个模块要解决的核心问题

用户按下"开始"，APP 要在**整个徒步过程（4–8 小时、可能锁屏、可能没信号、可能被系统杀进程）**里，持续、可靠地把 GPS 点记下来，并实时算出距离、爬升等数据；走完按"结束"，得到一条完整、可保存可导出的轨迹。

三条底线：
1. **不丢点**：后台、锁屏、息屏都要持续记录。
2. **不丢轨迹**：APP 崩溃/被杀/没电关机后，重启能恢复未结束的记录。
3. **电量可控**：全程记录不能把电耗光。

---

## 2. 记录的生命周期与状态机

```
空闲 ──开始──▶ 记录中 ──暂停──▶ 已暂停
                 ▲                  │
                 └──────继续────────┘
记录中/已暂停 ──结束──▶ 结算保存──▶ 空闲
任意进行中状态 ──APP被杀/崩溃/关机──▶（重启后）恢复进行中状态
```

| 状态 | 说明 |
|------|------|
| 空闲 | 未在记录 |
| 记录中 | 持续采集 GPS 点、累加统计、前台常驻通知 |
| 已暂停 | 停止采点（休息/午餐），保留已记录数据，统计冻结 |
| 结算保存 | 计算最终统计、生成轨迹、存入轨迹库 |

**手动暂停 vs 自动暂停**：手动暂停为主；**自动暂停进 MVP（已定）**——检测到长时间静止自动暂停、移动后自动继续。

---

## 3. 实时数据面板

| 数据 | 优先级 | 说明 |
|------|:---:|------|
| 累计距离 | P0 | 相邻点球面距离累加（含去噪后） |
| 总用时 / 运动用时 | P0 | 总用时含暂停；运动用时不含暂停 |
| 当前海拔 | P0 | GPS 海拔，或气压计（若有，更准） |
| 累计爬升 / 下降 | P0 | 海拔上升段之和 / 下降段之和（需去噪，见第 6 节） |
| 实时 / 平均速度、配速 | P1 | |
| 已记录点数 / GPS 精度状态 | P0 | 让用户知道定位好不好 |

---

## 4. 采样策略（精度 vs 电量的平衡）

- **采样方式**：按时间结合最小位移过滤。
- **MVP 默认值（已定）**：**每 5 秒采一个点 + 最小位移 5m**（位移小于 5m 视为静止漂移，不计入距离）。
- P1：提供"省电 / 均衡 / 高精度"几档预设，或让用户自定义。
- **静止时降频**：检测到基本不动时降低采样，省电并减少原地漂移产生的噪点。

---

## 5. 后台记录（平台难点，P0）

徒步时手机常在兜里、锁屏。必须保证后台持续定位（**目标平台：仅 iOS**）：

- **iOS 后台定位**：开启 Background Modes → Location updates；申请 **`Always`（始终允许）** 定位权限。
- **持续后台运行**：使用持续后台定位（如 `allowsBackgroundLocationUpdates`），并显示蓝色状态条/灵动岛提示，告知用户正在后台定位。
- **活动实时提示**：可用 Live Activity / 锁屏小组件显示"正在记录 + 距离/时长"，便于随时查看与暂停/结束（P1）。
- 权限引导：清晰的 `Always` 权限申请文案，说明为何需要后台定位。

> iOS 后台定位行为统一、可预期，比 Android（厂商保活）省心。**Android 不在当前范围**，若未来扩展再单独处理保活/前台服务/电池白名单。
> 这是徒步 APP 的"生死线"，技术选型时仍要重点验证后台定位的长时间稳定性与耗电。

---

## 6. 数据质量：去噪与海拔处理

GPS 原始数据有噪声，直接累加会让距离/爬升虚高。

- **位置去噪**：丢弃精度过差的点（accuracy > 阈值）；最小位移过滤（位移小于 X 米视为静止漂移，不计距离）。
- **海拔去噪**：海拔抖动大，累计爬升要用平滑/阈值，**默认单次海拔变化 < 5m 不计入累计爬升（已定）**，否则原地不动也会"爬升几百米"。
- **气压计（已定）**：**MVP 即接入**——设备有气压传感器时用气压计海拔（更稳，作优先源）；无传感器时回退 GPS 海拔。
- 原则：**存原始点**，统计值由原始点按规则计算，便于日后改进算法重算。

---

## 7. 崩溃恢复（P0，不丢轨迹的关键）

- 记录过程中**持续增量落盘**（每采到点/每隔几秒写入本地持久化），不是只存内存、结束才写。
- APP 启动时检查是否存在"未结束的记录会话"：
  - 有 → 提示"检测到上次未完成的记录，是否恢复？"，恢复后可继续记录或直接结算保存。
- 落盘要扛得住进程被杀、闪退、关机。

---

## 8. 航点 / 标记（P1）

- 记录过程中可打**航点**：当前位置 + 名称 + 类型（营地/水源/岔路/危险/拍照点）+ 备注。
- 拍照航点（关联照片）为 P1/P2。

---

## 9. 结束与保存

- 按"结束" → 计算最终统计 → 生成轨迹缩略图（地图小图）→ 存入本地轨迹库。
- 可在保存时编辑名称、备注。
- 异常极短轨迹（如 0 点 / 几米）提示用户是否丢弃。

---

## 10. 边界情况清单

| 情况 | 期望行为 |
|------|----------|
| GPS 长时间无信号（峡谷/山洞） | 不报错，标记定位差；恢复后续记；**中间断开的两段以虚线"推测段"连接**（已定），不当作真实行进里程 |
| 进程被系统杀死 | 已落盘数据不丢，重启可恢复（见第 7 节） |
| 电量耗尽关机 | 同上，靠增量落盘 |
| 长时间静止（休息） | 降频采样，不堆积漂移噪点；可自动暂停（P1） |
| 权限被中途撤销 | 检测到无定位权限，醒目提示并暂停记录 |
| 存储空间不足 | 提示用户，避免写入失败丢数据 |
| 边记录边导航 | 与导航模块联动：实走线 = 本模块记录的轨迹 |

## 11. 数据模型（与 PRD 第 6 节、GPX 对齐）

```
RecordingSession（进行中会话，用于崩溃恢复）
  ├─ 状态(记录中/已暂停), 开始时间, 累计统计快照
  └─ 已采集 Points[]（增量落盘）

Track（结束后生成）
  ├─ 名称, 来源=记录, 创建时间
  ├─ 统计: 总距离, 总用时/运动用时, 累计爬升/下降, 最高/最低海拔
  └─ Segments[] → Points[]（经纬度, 海拔, 时间戳, 精度, 速度?）

Waypoint[]（可选，记录中打的点）
```

---

## 12. 本模块决策记录（全部已定）

- ✅ **MVP 采样参数**：每 5 秒一个点 + 最小位移 5m。
- ✅ **自动暂停进 MVP**：静止自动暂停、移动自动继续。
- ✅ **海拔来源**：MVP 即接入气压计（可用时优先），无则回退 GPS 海拔。
- ✅ **爬升去噪阈值**：单次海拔变化 < 5m 不计入累计爬升。
- ✅ **无信号断段**：两段间以虚线"推测段"连接，不计入真实里程。
