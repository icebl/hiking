# 徒步 APP 需求文档（PRD）

> 版本：v0.1（初稿）
> 日期：2026-06-24
> 状态：需求确认阶段，技术栈待定
> 对标产品：两步路户外助手、六只脚

---

## 1. 产品概述

一款面向户外徒步用户的轨迹记录与导航 APP，核心价值是：**离线可用**、**记录自己的轨迹**、**沿别人/自己的轨迹徒步**、**看清地形（距离 + 等高线）**。

与对标产品的关系：
- **两步路**：行业标杆，功能全（轨迹库、社区、离线包、轨迹导航、海拔图）。我们做其核心子集。
- **六只脚**：偏轻量、轨迹分享起家。我们起步阶段更接近它（文件分享、本地优先）。

**起步定位**：先做一个"个人离线徒步工具"（本地优先、无需账号也能用），再逐步加云端社区能力。

---

## 2. 目标用户与典型场景

| 用户 | 场景 |
|------|------|
| 徒步/登山爱好者 | 进山前下载好离线地图和等高线，全程无信号也能记录和导航 |
| 轨迹跟随者 | 拿到别人分享的 GPX 轨迹，按这条线走，偏航时收到提醒 |
| 轨迹分享者 | 走完一条线，导出 GPX/分享给同伴 |

**关键约束**：山里**没有网络信号**。所以"离线"不是加分项，是 P0 底线——地图、等高线、轨迹、导航全部要能离线工作。

## 3. 功能需求（按模块 + 优先级）

优先级：**P0 = MVP 必须**，**P1 = 第二期**，**P2 = 后续**。

### 3.1 轨迹记录（P0）

- **开始/暂停/继续/结束** 记录。
- **后台持续记录**：锁屏、切到后台仍记录（这是徒步 APP 的生死线，需处理各平台后台定位与省电策略）。
- 实时数据面板：
  - 累计**距离**（P0）
  - 用时 / 运动用时（P0）
  - 当前海拔、累计**爬升 / 下降**（P0）
  - 实时速度 / 平均速度 / 配速（P1）
- 记录点采样策略：按时间（如每 1–3 秒）或按距离采样，可配置，平衡精度与电量（P1 可配置，P0 用固定默认值）。
- 轨迹中途添加**航点/标记点**（拍照点、营地、岔路口）（P1）。
- 异常恢复：APP 被系统杀死后，重启能恢复未结束的记录（P0，避免丢轨迹）。

### 3.2 沿轨迹徒步 / 轨迹导航（P0）

- 加载一条已有轨迹（自己的或导入的），在地图上显示这条**计划线**。
- 显示**我当前位置**与计划线的关系：方向、距线距离。
- **偏航提醒**：偏离轨迹超过阈值（如 50m）震动/声音提醒（P0，徒步导航的核心）。
- 显示前方剩余距离、剩余爬升（P1）。
- 沿线的航点提示（接近岔路口/水源点时提示）（P1）。

> 注：这里是"轨迹跟随"，不是道路级转向导航。徒步多在无路网区域，跟线即可，不需要像驾车导航那样算路。

### 3.3 离线地图（P0）

- 在有网时**按区域下载离线地图包**（框选范围 + 选择缩放级别）。
- 离线状态下完整浏览地图：平移、缩放、显示轨迹。
- 地图数据源方案（**关键技术决策，见第 5 节**）。
- 离线包管理：查看已下载区域、占用空间、删除（P1）。

### 3.4 等高线与海拔（P0）

- 地图上叠加**等高线**（P0）。
- 轨迹**海拔剖面图**：横轴距离、纵轴海拔，点击图表联动地图位置（P0）。
- 坡度显示（P1）。
- 等高线数据同样需要**离线**可用（见第 5 节，这是难点）。

### 3.5 导入轨迹（P0）

- 导入 **GPX**（P0，户外通用格式）。
- 导入 **KML/KMZ**（P1）。
- 来源：从文件系统、从微信/其他 APP "用本应用打开"、系统分享面板接收（P0 至少支持文件打开）。
- 导入后存入本地轨迹库，可查看、可作为导航线。

### 3.6 分享 / 导出（P0，文件方式）

- 导出轨迹为 **GPX**（P0）/ KML（P1）。
- 通过系统分享面板发给微信/邮件/其他 APP（P0）。
- 生成轨迹分享图（带海拔图、距离统计的图片）（P2）。
- 云端链接分享 / 二维码分享（P2，依赖后端）。

### 3.7 轨迹库管理（P0）

- 本地轨迹列表：缩略图、距离、爬升、日期。
- 轨迹详情：地图 + 海拔图 + 统计数据。
- 重命名、删除、分组/标签（P1）。

### 3.7.1 首页活动 · 公告区（新增，轻量联网）

- 首页展示**运营/官方发布的活动信息与公告**（Banner + 列表）。
- 内容由服务端发布，APP 联网拉取；**MVP 作为可选联网模块**：有网显示（缓存最近一次），无网隐藏/显示缓存。
- 依赖一个**轻量内容接口**（可早于完整后端先行；或并入三期 `End`）。纯徒步核心功能不依赖它。

### 3.8 账号与云端（第三期，已确认要做）

> 用户已明确：后续 APP **需要注册和登录**，且**轨迹可以上传到云端**。以下为已确认的三期能力，非"可能做"。

- **账号注册 / 登录**（已确认）。
- **轨迹上传云端 + 多设备同步 / 备份**（已确认）。
- 轨迹广场 / 搜索附近轨迹 / 热门路线（社区，规划中）。
- 云端链接分享 / 二维码分享（依赖后端）。
- 关注、评论等社区功能（规划中）。
- 对应你的 `End`（后端）目录。**MVP（一期）仍不做云端、纯本地优先**；但因云端已确认，MVP 的本地数据层需为云同步预留（见第 8 节决策）。

## 4. 非功能需求

| 项 | 要求 |
|----|------|
| 离线 | 地图、等高线、轨迹、记录、导航全部离线可用（P0） |
| 后台定位 | 长时间后台记录不被系统杀死，需处理保活与权限引导 |
| 电量 | 全程记录（4–8 小时）电量可控，采样频率可调 |
| 定位精度 | 利用 GPS + 可选融合传感器；记录原始轨迹，提供轨迹平滑/去噪 |
| 权限 | 定位（前台+后台）、存储/文件、（可选）传感器；需合规的权限申请文案 |
| 数据安全 | 本地轨迹不丢失；导入导出格式标准化（GPX 1.1） |
| 兼容 | **仅 iOS**；最低 iOS 版本待技术选型确定 |

## 5. 关键技术难点：离线地图 + 离线等高线

这是整个产品**最硬的两块**，需求阶段先点明，方便后续选型。

### 5.1 离线地图
- 方案 A：**矢量瓦片**（MapLibre + MBTiles/PMTiles），离线包小、可换样式，推荐。
- 方案 B：栅格瓦片缓存（下载现成地图瓦片离线存储），简单但包大、有版权/接口限制。
- 数据源候选：OpenStreetMap 衍生数据（开源、可商用，需注意署名）、或国内地图厂商离线 SDK。
- ✅ **已定**：底图走 **OSM 系自建矢量瓦片**（坐标系统一 WGS-84）；卫星影像图层在线提供（离线仅 T_Google，含发布前版权待解）。详见模块-离线地图与等高线。

### 5.2 离线等高线（最难）
等高线不是地图自带的，需要**高程数据（DEM）**生成：
- 高程数据源：开放 DEM（如 SRTM 30m、Copernicus DEM 30m），免费、全球覆盖。
- 处理流程：DEM → 用工具（如 GDAL `gdal_contour`）生成等高线矢量 → 切成矢量瓦片 → 打进离线包。
- 海拔剖面：轨迹各点采样 DEM 高程，或直接用记录的气压/GPS 海拔。
- ✅ **已定**：等高线 DEM 首选 **Copernicus GLO-30**；有网=预生成下发+动态生成，无网=基于本地 DEM 动态生成。

> 建议：MVP 阶段先用 **OSM 矢量瓦片 + 开放 DEM 生成等高线** 的开源路线，成本可控、完全离线、无第三方接口依赖。

## 6. 核心数据模型（GPX 对齐）

```
Track（轨迹）
  ├─ id, 名称, 创建时间, 来源(记录/导入)
  ├─ 统计: 总距离, 总用时, 累计爬升, 累计下降, 最高/最低海拔
  └─ Segments[]（轨迹分段）
        └─ Points[]（轨迹点）
              ├─ 经度, 纬度
              ├─ 海拔(elevation)
              ├─ 时间戳
              └─ (可选) 速度/精度

Waypoint（航点）
  ├─ 经纬度, 海拔, 名称, 类型(营地/水源/岔路…)
  └─ 关联轨迹id(可选), 备注, 照片(P1)
```
GPX 文件天然就是 `trk/trkseg/trkpt` + `wpt`，数据模型与导入导出保持一一对应，减少转换损耗。

## 7. 版本规划（建议分期）

### MVP（第一期）— 个人离线徒步工具
目标：**一个人进山，能记录、能跟线、能离线看图看等高线、能导入导出。**
- 轨迹记录（含后台 + 崩溃恢复）
- 离线地图浏览 + 区域下载
- 等高线叠加 + 海拔剖面图
- 沿轨迹导航 + 偏航提醒
- GPX 导入 / 导出 / 系统分享
- 本地轨迹库

### 第二期 — 体验增强
- KML/KMZ、航点拍照、坡度、离线包精细管理
- 轨迹平滑去噪、更多实时数据
- 分享图生成

### 第三期 — 账号与云端（启用 `End` 后端，已确认要做）
- **账号注册/登录**、**轨迹上传云端 + 多设备同步/备份**（已确认）
- 轨迹广场、链接/二维码分享、社区互动（规划中）

## 8. 待决策清单（对账后更新）

**已解决（由模块细化决策收口）：**
- ✅ 地图数据源：OSM 系自建矢量底图（WGS-84）。
- ✅ 高程/等高线：Copernicus GLO-30 DEM + 有网预生成/动态、无网动态。
- ✅ 离线最高级别 z16；瓦片混合方案（卫星栅格 + 底图/等高线矢量）。
- ✅ 本地 COROS 瓦片：私有 Garmin IMG 格式，放弃复用。

**平台与云端预留（已定）：**
- ✅ **目标平台：仅 iOS**（避开 Android 厂商保活难题；后台定位行为统一可预期）。最低 iOS 版本待技术选型确定。
- ✅ **MVP 为云同步预留数据层**：MVP 不建后端、不写接口，但本地数据层提前预留同步字段（稳定轨迹 ID、创建/修改时间戳、软删除标记、是否已同步标志），三期接云改动最小。

**技术选型（已定稿，详见 [技术选型.md](技术选型.md)）：**
- ✅ 框架：原生 Swift + SwiftUI；地图：MapLibre Native iOS。
- ✅ 离线格式：PMTiles（矢量底图/等高线）+ 栅格缓存（卫星）。
- ✅ 本地库：SQLite + GRDB；定位/传感器：Core Location + CoreMotion；GPX：CoreGPX。
- ✅ 数据生成工具链：planetiler + GDAL gdal_contour + tippecanoe → PMTiles。

**暂挂（发布前再解）：**
5. 第三方卫星图层（T_Google）离线缓存的版权 / 替代授权影像方案。
